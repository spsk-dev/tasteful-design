# Research Summary: v1.1.0 Flow Audit Stack

**Project:** SpSk (Simple Skill) -- v1.1.0 Flow Audit milestone
**Domain:** SPA flow navigation, screenshot capture, HTML report generation, animation detection
**Researched:** 2026-03-28
**Overall confidence:** HIGH

## Executive Summary

The v1.1.0 flow audit feature adds SPA multi-step navigation with per-screen design review, producing a self-contained HTML diagnostic report. The stack additions are minimal and deliberate: Playwright MCP server for stateful browser control (replacing ad-hoc CLI screenshot commands), base64-encoded screenshots in a bash-generated HTML report (zero new dependencies), and the Web Animations API via `browser_evaluate` for runtime animation detection.

The critical architectural decision is **Playwright MCP over CLI scripts** for flow navigation. The existing `/design-review` uses `npx playwright screenshot <url>` for single-page captures, which is stateless. Flow audit requires clicking through an SPA -- persistent browser state, sequential actions, intermediate screenshots. Playwright MCP exposes individual tools (`browser_navigate`, `browser_click`, `browser_take_screenshot`, `browser_evaluate`) that Claude calls sequentially with the browser staying open between calls. This is the correct model for multi-step SPA interaction.

The HTML report is generated by a bash script (`generate-report.sh`) that reads a JSON manifest and a screenshots directory, producing a single self-contained HTML file with inline CSS, base64-encoded images, expandable per-screen sections, and SpSk branding. No templating engine, no Node.js, no npm dependencies. This maintains the plugin's zero-dependency principle.

Animation detection uses standard browser APIs (Web Animations API via `Element.getAnimations()` and `getComputedStyle()`) injected via Playwright MCP's `browser_evaluate` tool. This provides both declared animations (from CSS) and actually-running animations (from runtime), addressing the existing v1.0 Motion specialist's limitation of source-code-only analysis.

## Key Findings

**Stack:** Playwright MCP (`@playwright/mcp`) for browser automation, bash heredoc for HTML report generation, Web Animations API for runtime animation detection. Zero new npm dependencies.

**Architecture:** Flow audit orchestrator (`/design-audit` command) drives Playwright MCP tools sequentially, dispatches existing 8-specialist review per screen, feeds results to `generate-report.sh`.

**Critical pitfall:** `networkidle` wait strategy hangs on real SPAs with analytics/WebSockets. Use `waitForURL` + element waits exclusively.

## Implications for Roadmap

Based on research, suggested phase structure for v1.1.0:

1. **Flow Navigation Engine** - Playwright MCP integration + SPA click-through
   - Addresses: Multi-screen navigation, intent parsing, screenshot capture
   - Avoids: networkidle hang, SPA pushState detection failure, screenshot timing issues

2. **Per-Screen Review Integration** - Wire existing 8-specialist review into flow context
   - Addresses: Per-screen specialist review, cross-screen consistency
   - Avoids: Token explosion (use quick mode for intermediate screens)

3. **HTML Report Generator** - `generate-report.sh` with base64 screenshots
   - Addresses: Self-contained HTML report, branded output, expandable sections
   - Avoids: Base64 size explosion (JPEG compression, lazy loading, size budget)

4. **Animation Detection** - Runtime + static hybrid via browser_evaluate
   - Addresses: Transition detection, cross-screen motion analysis
   - Avoids: Static-only analysis missing CSS-in-JS and conditional animations

**Phase ordering rationale:**
- Navigation engine first because everything depends on captured screens
- Per-screen review second because report needs scores to display
- Report generator third because it consumes all prior outputs
- Animation detection last because it is additive, not blocking

**Research flags for phases:**
- Phase 1: Needs Playwright MCP integration testing (MCP not yet registered in this environment). Test with a real SPA.
- Phase 3: HTML template design needs iteration. Start simple, add visual complexity after core works.
- Phase 4: Animation detection is well-understood technically but may need tuning for real-world SPAs.

## Gaps to Address

- **Playwright MCP not yet registered** in this environment. Need `claude mcp add playwright -- npx @playwright/mcp@latest` before implementation.
- **Report template design** not finalized -- need to iterate on layout during implementation.
- **Cross-platform base64 encoding** -- macOS vs Linux `base64` flags differ. Script must handle both.
- **Flow audit eval fixtures** -- need static SPA test apps for reproducible eval runs. Consider a simple multi-route HTML app bundled in `evals/fixtures/`.
