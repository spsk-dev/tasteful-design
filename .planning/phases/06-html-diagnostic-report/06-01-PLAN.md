  <name>Task 1: Create generate-report.sh with HTML template, screenshot embedding, and flow map</name>
  <files>scripts/generate-report.sh</files>
  <action>
Create `scripts/generate-report.sh` -- a bash script that reads flow-state.json and outputs a self-contained HTML report. Zero npm dependencies per D-91.

**Arguments:**
- $1: path to flow-state.json (required)
- $2: output HTML path (optional, defaults to same directory as flow-state.json with name `report-{timestamp}.html`)

**Script structure:**

1. **Validate input:** Check flow-state.json exists, is valid JSON (via jq), has screens array. Exit 1 with clear error if not.

2. **Screenshot processing (per D-104, REPT-07):**
   For each screen's screenshot_path:
   - Convert PNG to JPEG at 80% quality, max 1200px width using `sips` (macOS) with fallback to ImageMagick `convert` if sips unavailable
   - Base64 encode the JPEG: `base64 < file.jpg` (macOS) or `base64 -w0 < file.jpg` (Linux) -- detect platform via `uname`
   - Store as data URI: `data:image/jpeg;base64,{encoded}`
   - Track cumulative size. If over 4MB after all screenshots, re-compress at lower quality (60%) and retry

3. **HTML generation via heredoc:**
   Use `cat << 'REPORT_EOF'` with variable interpolation for the dynamic parts. The HTML structure:

   ```
   <!DOCTYPE html>
   <html lang="en">
   <head>
     <meta charset="UTF-8">
     <meta name="viewport" content="width=device-width, initial-scale=1.0">
     <title>SpSk Flow Audit -- {flow_intent or url}</title>
     <style>/* all CSS inline */</style>
   </head>
   <body>
     <header>SpSk signature + flow metadata</header>
     <section id="flow-map">Flow map with thumbnails</section>
     <section id="summary">Overall flow score + top 5 fixes</section>
     <section id="screens">Per-screen sections</section>
     <section id="consistency">Cross-screen consistency</section>
     <section id="animation">Animation summary</section>
     <footer>SpSk footer</footer>
   </body>
   </html>
   ```

4. **CSS design (per D-100, D-101, D-102, D-103):**
   - Dark theme: `--bg: #18181b` (zinc-900), `--surface: #27272a` (zinc-800), `--border: #3f3f46` (zinc-700), `--text: #e4e4e7` (zinc-200), `--text-muted: #a1a1aa` (zinc-400), `--accent: #3b82f6` (blue-500)
   - Score bars: CSS `<div>` with percentage width, accent color fill, zinc-700 track -- NOT Unicode blocks
   - Responsive: single column on mobile (<768px), max-width 1200px centered on desktop
   - Cards for each screen section with subtle border and surface background
   - SpSk signature in header, repo link in footer per D-102

5. **Flow map section (per D-95, REPT-02):**
   - Horizontal scrollable strip showing screen thumbnails (120px wide) connected by arrows
   - Each thumbnail: small screenshot, screen number, name below, score badge (color-coded: green >= 7.5, yellow >= 5.0, red < 5.0)
   - Clicking a thumbnail scrolls to that screen's detail section (anchor link)

6. **Summary section (per D-98, REPT-04):**
   - Overall flow score as large CSS bar with verdict badge (SHIP green, CONDITIONAL yellow, BLOCK red per D-99)
   - Flow metadata: screen count, mode, intent, started/completed timestamps
   - Top 5 priority fixes (from flow_score, ranked by severity/frequency)
   - Consistency summary line and animation compliance line

7. **Per-screen sections (per D-96, REPT-03, REPT-05):**
   For each screen in flow-state.json:
   - Full-width embedded JPEG screenshot (base64 data URI per REPT-01)
   - Score overview: overall screen score bar + verdict
   - Specialist scores: horizontal bar chart for each specialist (8 or 4 depending on review mode). Each bar labeled with specialist name, score value, filled bar
   - Specialist details inside `<details><summary>` elements (collapsed by default per D-97, REPT-05). Summary shows specialist name + score. Details show the specialist's findings and recommendations
   - Issues list with severity icons (matching shared/output.md symbols adapted for HTML)
   - Fix recommendations for this screen

8. **Consistency section:**
   - Only rendered if consistency findings exist
   - Finding cards with severity color coding (critical=red, mismatch=yellow, drift=blue)
   - Shows which screens are affected vs reference
   - Penalty calculation displayed

9. **Animation section:**
   - PRM compliance badge (PASS green, PARTIAL yellow, FAIL red)
   - Animation findings list
   - Duration/easing quality notes

10. **Print styles (per D-106, REPT-06):**
    ```css
    @media print {
      body { background: white; color: black; }
      details { display: block; } /* Force all expanded */
      details > summary { display: none; } /* Hide toggle controls */
      .flow-map { page-break-after: always; }
      .screen-section { page-break-inside: avoid; }
      .no-print { display: none; }
    }
    ```

11. **JavaScript (minimal, inline):**
    - Smooth scroll for flow map thumbnail clicks
    - "Expand all / Collapse all" toggle button for specialist details
    - No external dependencies

**Cross-platform considerations:**
- `sips` for macOS image conversion, `convert` (ImageMagick) for Linux fallback
- `base64` flag differences: macOS uses no flag, Linux uses `-w0` for no line wrapping
- `jq` is the only external dependency (already used by evals)
- `date` format differences: use `date -u +%Y%m%dT%H%M%SZ` (works on both)

**Error handling:**
- Missing screenshots: show placeholder "Screenshot not available" with grey box
- Incomplete flow-state.json (no review data): show navigation-only report with screenshots and flow map, skip score sections
- Missing jq: exit with "jq is required. Install with: brew install jq (macOS) or apt install jq (Linux)"

Make the script executable: `chmod +x scripts/generate-report.sh`
