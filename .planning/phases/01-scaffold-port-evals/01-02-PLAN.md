    **Required sections:**

    1. **Overview**: One paragraph explaining the multi-agent design review system. What problem it solves, why multi-agent beats single-agent.

    2. **System Architecture Diagram** (ASCII): Show the flow:
       User -> /design-review -> Boss Synthesizer -> 8 Specialists (parallel) -> Weighted Scoring -> SHIP/BLOCK Verdict

    3. **The 8 Specialists**: Table with each specialist's name, what dimension they score (1-10), what they look for, and their weight in the final score. Extract the exact specialist names, dimensions, and weights from `commands/design-review.md` and `config/scoring.json`. Include all 8.

    4. **Boss Synthesizer Pattern**: Explain how the boss agent:
       - Receives all 8 specialist scores + rationale
       - Applies weighted scoring formula (document the exact formula from scoring.json)
       - Produces final score and SHIP/BLOCK verdict
       - WHY: This pattern was chosen because single-agent review scored 40% -- specialists catch domain-specific issues a generalist misses

    5. **Scoring Algorithm**: Document the exact formula. Reference the weights from scoring.json. Explain:
       - How dimension weights are applied
       - What the SHIP threshold is
       - How quick mode (`--quick`) changes the formula (4 specialists, /13 weights per PORT-10)

    6. **Degradation Tiers**: Table showing:
       - Tier 1: Claude + Gemini CLI + Haiku -- full capability, all specialists, cross-model validation
       - Tier 2: Claude-only -- no Gemini, reduced confidence, still functional
       - Tier 3: Code analysis only -- structural checks, no AI review
       - For each tier: what triggers it, what's available, what's lost
       - WHY: Graceful degradation means the tool always works, never errors out

    7. **Anti-Slop System**: Explain what anti-slop.json does -- detecting and penalizing common AI-generated design patterns. List 3-5 example patterns from the actual anti-slop.json file.

    8. **Command Architecture**: Brief overview of the 4 commands and how they relate:

       - /design-review: the core review
       - /design-improve: iterative build/review/fix loop
       - /design-validate: functional testing via Playwright

    9. **Design Decisions**: Table of key decisions with "Decision", "Alternative", "Why This Way" columns. Include at minimum:
       - Multi-agent over single-agent (40% vs 100%)
       - Markdown commands over JavaScript (no build step, commands ARE prompts)
       - JSON config over hardcoded values (tunable without prompt editing)
       - Weighted scoring over simple average (some dimensions matter more)
       - Graceful degradation over hard requirements (always works)

    10. **File Map**: Brief directory listing showing where each component lives in the repo.

    Tone: Technical but accessible. A VP of Engineering should understand the architecture. A senior developer should understand the implementation. Avoid jargon without explanation.

    Create CHANGELOG.md (per D-13, SCAF-05) that tells the story of iteration from failure to success. This is a differentiator -- most projects hide failures.

    **Required structure (Keep Changelog format):**

    ```
    # Changelog

    All notable changes to the SpSk design-review plugin.

    The format is based on [Keep a Changelog](https://keepachangelog.com/).

    ## [1.0.0] - 2026-03-28

    ### What Works
    - 8-specialist multi-agent review achieving 8.6/10 consensus score
    - Weighted scoring with SHIP/BLOCK verdict
    - 3-tier degradation (full, Claude-only, code-only)
    - Quick mode (4 specialists, faster reviews)
    - Anti-slop detection for AI-generated patterns
    - Reproducible eval harness with range-based assertions

    ### The Journey (What Didn't Work)

    **v1 - Single Agent (Score: ~40%)**
    - One Claude instance tried to review everything
    - Missed domain-specific issues (typography, color theory, motion design)
    - Reviews were generic and unhelpful
    - Lesson: Generalists produce generalist feedback

    **v2 - Specialist Agents Without Coordination**
    - Split into specialists but no synthesis
    - Contradictory feedback between specialists
    - No clear verdict or priority
    - Lesson: Specialists need a boss

    **v3 - Boss + Specialists, Unweighted**
    - Added boss synthesizer to reconcile specialist feedback
    - Equal weighting produced misleading scores
    - Minor issues weighted same as critical problems
    - Lesson: Not all dimensions are equally important

    **v4 - Weighted Multi-Agent (Current)**
    - 8 weighted specialists + boss synthesizer
    - Scoring formula reflects real design priorities
    - SHIP/BLOCK verdict with clear threshold
    - Consensus score: 8.6/10 from 3-model review
    ```

    Adapt this structure using information from:
    - The source CHANGELOG.md (has v1.0.0 entry)
    - .context/design-review-status.md (has benchmark results)
    - .context/plan.md (has the 40% to 100% narrative)

    The version numbers (v1, v2, v3, v4) represent design iterations, NOT semver releases. Make this clear. The key narrative: transparency about failure builds more credibility than claiming perfection.
  </action>
