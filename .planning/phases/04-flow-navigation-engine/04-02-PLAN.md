Create `commands/design-audit.md` per D-56 through D-73. This is a prompt file (not a script) that instructs the agent to orchestrate Playwright MCP tools.

**Frontmatter** (per D-56). Use standard YAML frontmatter delimiters:
```yaml
#---
name: design-audit
description: >
  Flow-level design audit -- navigates multi-screen SPA flows using Playwright MCP,
  captures screenshots at each state change, and produces flow state for downstream
  review and reporting. Use when the user says "audit the flow", "design audit",
  "walk through the onboarding", "/design audit", or wants to evaluate a multi-screen
  user journey.
allowed-tools: Bash(mkdir *), Bash(cp *), Bash(rm *), Bash(cat *), Bash(npx *)
#---
```

**Structure the command body with these sections:**

**1. Imports:**
- `@${CLAUDE_PLUGIN_ROOT}/shared/output.md` for branded output
- `@${CLAUDE_PLUGIN_ROOT}/skills/design-review/references/flow.md` for navigation knowledge
- Reference `@${CLAUDE_PLUGIN_ROOT}/config/flow-scoring.json` for default values

**2. Argument Parsing (per D-57, D-58, D-59):**
Parse `$ARGUMENTS` for:
- URL -- required, the starting URL
- `--flow "description"` -- intent mode (mutually exclusive with --steps)
- `--steps url1,url2,url3` -- deterministic mode (comma-separated URLs)
- `--auth` -- authenticated flow mode
- `--max-screens N` -- override max screen limit (default from config: 10)
If neither --flow nor --steps provided, ask the user which mode they want.

**3. Setup:**
- Create output directory: `/tmp/design-audit-{timestamp}/` where timestamp is YYYYMMDD-HHMMSS (per D-68)
- Initialize flow-state.json with status "in_progress", the parsed arguments, empty screens array (per D-72)
- Show branded header: ` SpSk  design-audit  v1.1.0  --- flow navigation engine`

**4. Playwright MCP Verification:**
- Call `browser_navigate` with the starting URL
- If MCP tool not available, show error: "Playwright MCP not registered. Run: claude mcp add playwright -- npx @playwright/mcp@latest" and stop
- After successful navigation, call `browser_resize` to set viewport to 1440x900

**5. Authentication Handling (per D-66, for --auth mode):**
- After initial navigation, call `browser_snapshot`
- If snapshot shows login form indicators (input[type=password], "Sign In", "Log In" text), display a checkpoint message:
  "Login required. Please complete login in the browser window, then type 'continue' to resume the audit."
- Wait for user confirmation, then call `browser_snapshot` again to verify authenticated state
- If no login form detected and --auth was provided, proceed normally (already authenticated)

**6. Intent Mode Navigation Loop (per D-61, D-62, D-63, D-64):**
For each screen (up to --max-screens):
```
Step A: browser_snapshot -- read accessibility tree
Step B: Agent reasons about which CTA matches the flow intent
  - Look for buttons, links, form submit buttons that semantically match the intent
  - If multiple matching CTAs, pick the most prominent (primary button > secondary > link)
  - If no matching CTA found, check if this is a success/completion state (D-64)
  - If dead end (no CTAs, error page), terminate flow gracefully
Step C: browser_click ref="{identified_ref}" -- click the CTA
Step D: Run DOM stability check via browser_evaluate (800ms mutation quiet)
  - Use the MutationObserver pattern from references/flow.md
Step E: Run font readiness check via browser_evaluate (document.fonts.ready)
Step F: browser_snapshot -- confirm new screen has loaded (compare to pre-click snapshot)
Step G: browser_take_screenshot -- capture to /tmp/design-audit-{timestamp}/screen-{N}-{slug}.png
  - Generate slug per references/flow.md logic (h1/h2 > URL path > "screen-N")
Step H: Update flow-state.json -- append screen entry, write file (progressive persistence per Pattern 3)
Step I: Terminal output with branded format: "Screen {N}: {name} -- captured checkmark"
```

**7. Deterministic Mode Navigation (per D-65):**
For each URL in --steps list:
```
Step A: browser_navigate to URL
Step B: Run DOM stability check via browser_evaluate
Step C: Run font readiness check via browser_evaluate
Step D: browser_take_screenshot -- capture
Step E: Update flow-state.json -- append screen entry
Step F: Terminal output: "Screen {N}: {name} -- captured checkmark"
```
No CTA identification or clicking needed. Shared capture logic with intent mode.

**8. Flow Completion:**
- Update flow-state.json with status "complete", completed_at timestamp
- Show branded summary:
  - Total screens captured
  - Flow duration
  - Screenshot directory path
  - "Flow state: /tmp/design-audit-{timestamp}/flow-state.json"
- Show footer with next step hint: "Run Phase 5 review to analyze captured screens"

**9. Error Handling:**
- If a click fails (element not found): take fresh snapshot and retry once. If still fails, capture error screenshot, set flow status to "error" with error_at_screen, terminate gracefully.
- If DOM stability times out: proceed anyway (screen may still be usable), note in flow state
- If max screens reached (per D-59, FLOW-05): terminate with status "max_screens_reached", show warning

**Key constraints:**
- Do NOT import or dispatch specialist reviews -- that is Phase 5
- Do NOT generate HTML reports -- that is Phase 6
- The command is pure navigation + capture + flow state
- Every browser interaction uses Playwright MCP tools, never shell scripts
- Always take fresh browser_snapshot immediately before browser_click (stale ref protection per Pitfall 5)
- Never use networkidle (per Pitfall 1)
- The flow-state.json is the contract between Phase 4 and Phases 5/6 -- match the schema from RESEARCH.md exactly
